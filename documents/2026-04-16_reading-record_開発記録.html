1-2. ルート定義

routes/web.php を以下のようにグルーピングして記述:

Route::prefix('books')->name('books.')->controller(BookController::class)->group(function () {
    Route::get('/',              'index')->name('index');
    Route::get('/create',        'create')->name('create');
    Route::post('/',             'store')->name('store');
    Route::get('/{book}',        'show')->name('show');
    Route::get('/{book}/edit',   'edit')->name('edit');
    Route::put('/{book}',        'update')->name('update');
    Route::delete('/{book}',     'destroy')->name('destroy');
    Route::patch('/{book}/status', 'updateStatus')->name('updateStatus');
});

1-3. BookController 実装

index / create / store / show / edit / update / destroy / updateStatus を実装。

  • index: with('genre') + created_at DESC
  • show / edit / update / destroy: ルートモデルバインディングで Book $book
  • destroy: $book->delete() だけで論理削除（SoftDeletesトレイト）
  • updateStatus: ステータス即時更新用、JSONを返す

1-4. FormRequest に切り出し

sail artisan make:request StoreBookRequest
sail artisan make:request UpdateBookRequest

UpdateBookRequest extends StoreBookRequest とすることで、同じルールを使い回す構成に。

2. 学びと判断ポイント

2-1. ルートの順序に注意

books/create は books/{book} より上に書く必要がある。そうしないと create が {book} にマッチしてしまう。

2-2. RESTfulの考え方

  • URIは「リソース（モノ）」を表す
  • 動作（CRUD）はHTTPメソッドで表現する（GET/POST/PUT/DELETE）
  • 同じURI /books でも GET = 一覧、POST = 作成 というようにメソッドで振り分けられる
  • そのため /books/store のようにURIに動詞を入れる必要はない
メソッドURIアクション対象
GET/booksindexコレクション
GET/books/createcreateフォーム画面
POST/booksstoreコレクション
GET/books/{book}show個別リソース
GET/books/{book}/editeditフォーム画面
PUT/books/{book}update個別リソース
DELETE/books/{book}destroy個別リソース

2-3. ルートモデルバインディング

show(Book $book) のように型指定すると、URIの {book} から自動で該当レコードを取得（findOrFail相当）。見つからなければ自動で404。

2-4. N+1問題と Eager Loading

問題: 一覧で各bookのgenreを表示する時、with なしだと本の数だけSQLが発行される。
解決: with('genre') を付けると、使われている genre_id を WHERE IN (...) でまとめて1回で取得し、各bookに紐付けてくれる（Eager Loading）。
差: 本が10冊なら「11回のSQL → 2回のSQL」。

2-5. Enum の cases()

PHP の Enum には cases() が自動で用意されており、定義した全ケースを配列で返す。ビューでプルダウン生成に使う。将来ステータスを増やしても、Enumに1行足すだけでビューに反映される。

2-6. view() の引数

  • 第1引数はビューファイルのパス（resources/views/ 配下を . 区切り）。URLではない。
  • 第2引数はビューに渡すデータ（連想配列）。キーがビュー内の変数名になる。
  • ビューは渡されたデータしか使えないので、必要な値は全てコントローラで渡す。

2-7. バリデーションの場所

パターン特徴
コントローラ直書きシンプルだが、store/updateで重複しやすく肥大化
FormRequest（採用）責務分離・可読性・保守性・テスト容易性に優れる
モデル内Laravelでは非推奨。モデルは「データの状態」を担当する設計思想

実務主流の理由: 責務分離、可読性、保守性、テスト容易性、チーム規約としての予測可能性。

2-8. FormRequest の仕組み

  • コントローラメソッドが呼ばれる前にLaravelが自動でバリデーション実行
  • 失敗時は自動でリダイレクト＆エラーメッセージをセッション保存
  • コントローラは成功時の処理だけ書けばOK
  • $request->validated() で検証通過後のデータのみ取得（余計なパラメータは除外される）
  • authorize() で認可ロジックも統合可能
